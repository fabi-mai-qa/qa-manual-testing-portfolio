# Test Strategy

## Introduction
This document defines the overall testing principles and quality approach applied across the product to validate software quality through manual testing practices focused on user workflows, APIs, and data validation.

Testing follows a risk-based approach prioritizing critical business flows, integrations, and user-visible functionality.

## Objectives
- Apply a structured manual testing approach to validate functional behavior and system integrations.
- Detect defects early and reduce risks before delivery to end users.

## Scope of Testing

### In Scope:
- Manual functional testing.
- API testing (primarily REST, with exposure to SOAP when applicable).
- Smoke and exploratory testing.
- Database validation using SQL when required.
- Validation against defined acceptance criteria.

### Out of Scope:
- Test automation.
- Performance or load testing.
- Security penetration testing.
- Infrastructure testing.

## Test Approach
Testing combines:
- **Black box testing** from the user perspective.
- **Gray box testing** using knowledge of APIs, database data, and technical outputs generated by the system (such as validation messages, data records, or service responses) when needed.
- **Risk-based prioritization** focused on critical workflows and integrations.

QA collaborates with developers and product stakeholders to clarify expected behavior and reduce ambiguities before execution whenever possible.

## Test Environment and Data
Testing is performed in a dedicated QA/Test environment, which may include locally installed application components and remote services such as APIs or databases.

Test data is created manually or prepared in dedicated test databases.

## Entry and Exit Criteria

### Entry:
- Testable build available.
- Environment accessible.
- Acceptance criteria defined and clarified when necessary.

### Exit:
- Planned test cases executed.
- Critical defects resolved or assessed with stakeholders.
- Acceptance criteria validated through test results.

## Defect Management
- Defects are reported with reproduction steps, expected vs actual results, and supporting evidence such as screenshots, videos, technical messages generated by the system, or API responses.
- Issues are classified by severity and priority.
- QA collaborates with developers during investigation, fix validation, and release decisions.

## Test Deliverables and Metrics

### Deliverables:
- Test scenarios and test cases.
- Test summary report.
- Bug reports.

### Metrics:
- Test execution progress (passed vs failed test cases).
- Number of reported defects and their severity levels.
- Coverage of critical business flows within the test scope.

## Risks
- Test environment instability.
- Acceptance criteria unclear or incomplete.
